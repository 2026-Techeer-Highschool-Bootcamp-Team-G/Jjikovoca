# 찍어보카 문서 동기화 (doc-sync)

## 이 스킬이 푸는 문제

찍어보카의 설계 문서들은 **하나의 진실을 여러 각도에서 적은 계약**이다. 요구사항 F-번호가
ERD 테이블·컬럼을 낳고, 그게 API 요청/응답 필드가 되고, 사용자플로우의 화면 전환이 된다.
그래서 "단어에 발음 컬럼 추가" 같은 작은 변경도 보통 02·03·04 세 곳을 동시에 건드린다.
한 곳만 고치면 문서 간 계약이 깨지고, 나중에 구현할 때 어느 문서가 맞는지 알 수 없게 된다.

이 스킬의 일은 **연쇄 반영 대상을 빠짐없이 식별하고, 의존 순서대로, 불변 규칙을 지키며 반영**
하는 것이다.

(예: "이 F-XX는 Figma 14 화면과 Notion 요구사항 페이지에도 반영 필요").

## 워크플로우

### 1. 변경의 파급 범위를 먼저 그린다
반영을 시작하기 전에, 변경이 어느 문서를 건드리는지 **명시적으로 판정**하라. 판정표:

- 기능의 정의·조건·게이팅이 바뀌나 → **02**
- 저장되는 데이터(테이블/컬럼/타입/제약/관계)가 바뀌나 → **03**
- 클라이언트가 주고받는 필드·엔드포인트·에러가 바뀌나 → **04**
- 화면 전환·진입점·분기가 바뀌나 → **17**

하나만 해당하는 것처럼 보여도 한 번 더 의심하라. "컬럼 추가"는 03만 같지만, 그 컬럼이
API 응답에 나오면 04도, 사용자가 설정하는 값이면 02도 대상이다.

### 2. 의존 순서대로 반영한다
**02 → 03 → 04 → 17** 순서가 기본이다. 요구사항이 데이터 모양을 정하고, 데이터가 API
계약을 정하고, API가 화면 흐름을 정하기 때문이다. 앞 문서에서 내린 결정(필드명·enum 값·
기본값)을 뒤 문서가 그대로 인용하도록 하라 — 같은 것을 두 문서가 다르게 적으면 그게 버그다.

### 3. 불변 규칙을 지킨다 (반영할 때마다 자동)
1. **버전 증가 + 변경 이력** — 고친 문서마다 헤더 표의 버전을 올리고(vX.Y) 변경 이력에
   한 줄 추가. 날짜는 절대표기(예: 2026-07-23). 상대표현("오늘") 금지.
2. **sqlglot DDL 검증** — 03의 DDL을 바꿨으면 반드시 sqlglot MySQL 방언으로 파싱해
   구문이 깨지지 않았는지 확인. 절차·venv 경로는 `references/sync-rules.md`.

규칙 전문(반영 순서의 근거, 버전 규칙 세부, sqlglot 명령, 문서 우선순위, 비노출 계약 등
깨지기 쉬운 도메인 불변식)은 **`references/sync-rules.md`를 읽어라**.

### 4. 반영 요약을 보고한다
끝나면 **무엇을 어느 문서에 어떻게 고쳤는지** 문서별로 간결히 보고하라. 문서 전체를 되붙이지
말 것. 형식:

```
반영 완료 — <변경 한 줄 요약>
- 02 (v→vX.Y): <무엇을>
- 03 (v→vX.Y): <무엇을> · sqlglot 검증 ✓
- 04 (v→vX.Y): <무엇을>
- 17: <무엇을> (또는 "해당 없음")
다음 수동 단계: Figma <화면>, Notion <페이지> (있으면)
```

## 자주 하는 실수 (피하라)
- **한 문서만 고치고 끝내기** — 거의 항상 다른 문서와 계약이 어긋난다. 1단계 판정을 건너뛰지 마라.
- **필드명·enum을 문서마다 다르게 적기** — 02에서 `NUMERIC/CHOICE`로 정했으면 03·04도 똑같이.
- **버전·이력 누락** — 나중에 무엇이 언제 왜 바뀌었는지 추적 불가가 된다.
- **DDL 바꾸고 검증 생략** — 쉼표·CHECK·인덱스 구문 오류가 그대로 남는다.
- **Desktop 경로 접근 실패** — Bash가 /tmp에 갇히면 `dangerouslyDisableSandbox:true`, 편집은
  Read/Edit 도구로 절대경로 사용(`references/sync-rules.md`의 환경 주의 참조).
